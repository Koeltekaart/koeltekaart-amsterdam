# Heat Vulnerability Analysis — Amsterdam

This repository contains the data-analysis pipeline for the **Koeltekaart Amsterdam** capstone project. The `pipeline.ipynb` is a narrated and reproducible notebook that builds a neighbourhood-level **Heat Vulnerability Index (HVI)** for Amsterdam and exports the maps/tables used by the policy dashboard in the Amsterdam Heat Stress Website.

The analysis follows the IPCC risk framing: heat risk is not only about high temperatures. It is the combination of **heat exposure**, **social vulnerability**, and **capacity to access cooling**. The goal of the pipeline is therefore to identify where Amsterdam neighbourhoods are physically hot, socially vulnerable, and poorly served by cooling opportunities, so that urban-cooling interventions can be prioritised transparently.

## 3. Research questions

The pipeline answers four connected questions:

1. **RQ1 — Physical drivers of heat risk**  
   Which neighbourhood characteristics, such as impervious surface, public/private greenery, surface water, and tree canopy, explain official heat-risk scores?

2. **RQ2 — Social vulnerability and cooling-access disadvantage**  
   Are socially vulnerable neighbourhoods also worse served by cooling infrastructure or refuge-like amenities?

3. **RQ3 — Composite Heat Vulnerability Index**  
   Can heat exposure, social vulnerability, and cooling access be combined into a transparent and statistically robust HVI?

4. **RQ4 — Intervention prioritisation**  
   Where are the significant heat-vulnerability clusters, and which neighbourhoods should be prioritised for trees, shelters, or other cooling interventions?

## 4. How the pipeline works

### Step 1 — Environment and configuration

The notebook imports the scientific Python and spatial-analysis stack, resolves the repository root, creates the output folder, and defines helper functions for name cleaning, normalisation, scaling, and plotting.

Main packages used:

- `pandas`, `numpy`, `scipy`, `statsmodels`, `scikit-learn`
- `geopandas`, `libpysal`, `esda`, `mapclassify`
- `matplotlib`/`seaborn` for figures

### Step 2 — Load and join core data

The pipeline loads the Amsterdam buurt-level feature table, boundary geometries, and official Amsterdam climate-risk scores. Neighbourhood names are normalised before joining to reduce mismatches caused by accents, punctuation, or naming variants.

Expected core inputs:

| File | Role in the analysis |
|---|---|
| `inputs/ams_features.csv` | Base feature table containing CBS variables and precomputed remote-sensing/spatial features such as `temp_mean`, `ndvi_mean`, `water_prc`, and `road_prc`. |
| `inputs/buurten_geometry.geojson` | Buurt boundaries used for spatial joins, centroids, maps, and Queen-contiguity weights. |
| `inputs/climate_risk_scores.csv` | Official Amsterdam heat-risk and Hittestresskaart variables, including `HI_TOTAAL_S.0`, `HI_BLOOTSTELLING_S.0`, `HI_KO_PET_S.0`, and `HI_GEVOELIGHEID_S.0`. |
| `inputs/trees_raw.json` | Amsterdam street-tree register used to calculate tree density and canopy-related indicators. |
| `data/koeltekaart_data.csv` | Official/public cooling locations used to calculate the distance to the nearest koelteplek. |

### Step 3 — Engineer tree-canopy indicators

Street trees are converted to points in RD-New coordinates and spatially joined to buurt polygons. The notebook then calculates:

- `tree_count`
- `tree_density_per_km2`
- `pct_mature`, based on mapped tree-height classes
- `tree_species_richness`

This avoids relying on inconsistent tree-register neighbourhood labels and uses geometry instead.

### Step 4 — Build the cooling-access indicator

The notebook measures adaptive capacity in two ways:

1. **Refuge access** from CBS average distances to eight refuge-like amenities: swimming pools, libraries, large supermarkets, department stores, cinemas, museums, theatres, and ice rinks.
2. **Private cooling capacity** from the official access-to-garden score.

These components are normalised and averaged into `cooling_access`, where higher values mean better access. The distance to the nearest official koelteplek is kept separately as `dist_koelteplek_m` for a shelter-gap analysis; it is not used directly in the HVI because the official shelter list is too sparse to score every neighbourhood fairly.

### Step 5 — Exploratory analysis and diagnostics

Before modelling, the notebook checks distributions, normality, outliers, pairwise correlations, and multicollinearity. Important decisions made here include:

- using **Spearman** correlations where variables are non-normal or monotonic but not linear;
- using **HC3 robust standard errors** in OLS models;
- keeping policy-relevant extreme neighbourhoods in the sample while winsorising modelling variables at the 1st/99th percentiles;
- checking VIFs before interpreting regression coefficients.

### Step 6 — RQ1 model: physical drivers

Official heat risk (`HI_TOTAAL_S.0`) is regressed on physical-environmental predictors using OLS with HC3 robust errors. The model compares a baseline physical model with an expanded model that adds tree density.

The saved notebook output reports that adding tree density improves explanatory power, but the model still explains only part of official heat risk. This supports the assignment’s main argument: heat vulnerability cannot be reduced to physical heat exposure alone; population sensitivity and cooling access also matter.

### Step 7 — RQ2 model: social vulnerability and cooling access

The Social Vulnerability Index is built from:

- official heat sensitivity (`HI_GEVOELIGHEID_S.0`), weight 0.50;
- percentage of single-person households, weight 0.25;
- percentage of residents with a non-Western migration background, weight 0.25.

The notebook then compares this index with `cooling_access` using Spearman correlation, a high-vulnerability/low-access quadrant count, and quantile regression. This tests whether vulnerable neighbourhoods are also disadvantaged in access to cooling options.

### Step 8 — RQ3 model: composite HVI

The final HVI combines three 0–1 prongs:

\[
\text{HVI} = 0.40 \times \text{heat exposure} + 0.40 \times \text{social vulnerability} + 0.20 \times (1 - \text{cooling access})
\]

The prongs are defined as:

| HVI prong | Meaning | Main inputs |
|---|---|---|
| `heat_exposure` | Physical heat hazard | `temp_mean`, `HI_BLOOTSTELLING_S.0`, `HI_KO_PET_S.0` |
| `social_vuln` | Sensitivity of residents | `HI_GEVOELIGHEID_S.0`, single-person households, migration-background share |
| `1 - cooling_access` | Lack of adaptive capacity | refuge-distance index and access-to-garden score |

The HVI is validated and stress-tested through:

- convergent validity against the city’s official heat-risk score;
- ±10% weight-sensitivity analysis;
- bootstrapped Fisher-Jenks tier boundaries;
- Fisher-Jenks natural-break tiering into five HVI classes.

### Step 9 — RQ4 spatial clustering and intervention priority

The notebook tests whether high HVI values are spatially clustered using global Moran’s *I* and local LISA clusters with Benjamini-Hochberg FDR correction. It then creates an `intervention_priority` score from four percentile-ranked signals:

- high heat exposure;
- high social vulnerability;
- poor cooling access;
- low tree density.

It also flags **green deserts**: tier 4–5 neighbourhoods that are in the bottom quartile for both NDVI and tree density.

### Step 10 — Export results

At the end, the notebook writes:

| Output | Description |
|---|---|
| `outputs/hvi_scores.csv` | Per-buurt table with HVI, tier, prongs, priority class, green-desert flag, tree metrics, shelter distance, and official heat-risk score. |
| `outputs/hvi_dashboard.geojson` | Dashboard-ready GeoJSON with geometry, HVI scores, priority fields, and LISA cluster labels. |
| `outputs/fig_*.png` | Figures produced throughout the notebook for diagnostics, models, HVI distributions, maps, and shelter-gap analysis. |

## 7. Data provenance and reproducibility boundary

Everything downstream of `ams_features.csv` is reproduced in the notebook: joins, tree aggregation, feature engineering, indices, models, tiering, clustering, rankings, figures, and exports.

Some upstream spatial features are documented rather than regenerated inside the notebook:

| Dataset | Source | Used for | Rebuilt inside notebook? |
|---|---|---|---|
| CBS Kerncijfers wijken en buurten 2022 | Statistics Netherlands / PDOK | demographics, income, amenity distances | Loaded through `ams_features.csv` |
| Landsat 8 surface temperature | Google Earth Engine, summer 2023 median | `temp_mean` heat exposure | No, precomputed into `ams_features.csv` |
| Landsat 8 NDVI | Google Earth Engine, summer 2023 median | `ndvi_mean` greenness | No, precomputed into `ams_features.csv` |
| BGT water and road surface shares | PDOK/QGIS | `water_prc`, `road_prc` | No, precomputed into `ams_features.csv` |
| Amsterdam heat-risk scores | Gemeente Amsterdam / Klimaateffectatlas | official heat-risk and Hittestresskaart variables | Yes, loaded and joined from `climate_risk_scores.csv` |
| Street trees | Gemeente Amsterdam open data | tree density, maturity, species richness | Yes, spatially joined from `trees_raw.json` |
| Cooling spots | project koelteplek list | shelter-gap diagnostic | Yes, loaded from `data/koeltekaart_data.csv` |
| Buurt boundaries | CBS/PDOK buurt geometry | maps, centroids, spatial weights | Yes, loaded from `buurten_geometry.geojson` |

This boundary is important: the notebook is fully reproducible for the submitted analysis stage, but it does not claim to rebuild every remote-sensing or QGIS preprocessing step from raw web services.

## 8. Interpretation guide for markers

The assignment contribution is not only the final HVI map, but the reasoning chain that produces it:

- **RQ1** shows why physical exposure matters but is not sufficient.
- **RQ2** adds the equity dimension by testing whether vulnerable residents also face worse cooling access.
- **RQ3** combines hazard, sensitivity, and adaptive capacity into a transparent composite index.
- **RQ4** converts the index into policy guidance: clusters, priority neighbourhoods, green deserts, and shelter gaps.

The most important files for assessment are:

1. `pipeline.ipynb` — method, code, figures, and interpretation;
2. `outputs/hvi_scores.csv` — numerical result table;
3. `outputs/hvi_dashboard.geojson` — spatial layer for the dashboard/map;
4. this README — overview and reproduction guide.

## 9. Limitations

- Heat exposure is based on a summer-2023 daytime Landsat composite. The project does not include night-time heat, even though night-time heat is highly relevant for health.
- Several spatial variables are upstream inputs from GEE/QGIS workflows, so the notebook documents but does not regenerate those preprocessing steps.
- Cooling access is a proxy based on amenity distances and access to gardens. The official koelteplek list contains only twelve locations, so it is more useful as a shelter-gap diagnostic than as a full city-wide access index.
- HVI weights are a transparent judgement rather than a universal truth. The notebook mitigates this with ±10% sensitivity testing and bootstrapped tier boundaries.
- Some joins depend on normalised neighbourhood names. The notebook reports coverage so reviewers can check whether mismatches are material.
